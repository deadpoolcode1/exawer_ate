# M1 Acceptance Criteria

This document is the contract for M1 ("Document Parser") completion. The numeric thresholds below are the gate. The scorecard generated by `./modular_tools.sh verify` evaluates each metric against the threshold. M1 is shippable only when every line is `[PASS]`.

## TL;DR for the Project Owner

Run **one** command:

```
./modular_tools.sh verify
```

If the last line says `OVERALL: PASS`, M1 is ready for Exaware spot-check.
If it says `OVERALL: FAIL`, the named metric is below its threshold and M1 is not shippable.

## Metrics

| ID | Metric | Threshold | What it actually means |
|---|---|---|---|
| M1.a | heading_recovery | ≥ 95% | Detected headings match the committed golden list (manually reviewed). Catches regressions where heading detection silently drops sections. |
| M1.b | cli_block_preservation | = 100% | Each curated CLI signature in the EVPN spec appears verbatim in the IR's code blocks. Hard gate — these become test inputs in M3/M4. |
| M1.c | table_preservation | ≥ 90% | Each tracked document yields at least the expected number of structured tables. Tables come back as `rows × cells`, not flattened text. |
| M1.d | anchor_detection (M2) | reported | Count of `EVPNS-REQ#NN` anchors visible in IR full text. Reported, **not gated** at M1 — formal scoring is M2 territory. |
| M1.e | format_parity | word-Jaccard ≥ 0.90 | RFC 9785 in DOCX/TXT/PDF produces ≥ 90% identical word sets. Confirms "same content, three formats." |
| M1.f | determinism | 3/3 byte-identical | Three consecutive runs over the same input produce byte-identical IR. Catches non-determinism (sets-with-id, etc.) before it hides downstream bugs. |
| M1.g | no_unhandled_exceptions | = 0 | The parser raises no untyped exceptions on Tier-A/B corpus. |
| M1.h | performance | < 30 s | rfc7432bis-13.docx (450 KB) parses in under 30 seconds on dev hardware. |
| M1.i | edge_cases | manifest match | Each Tier-C edge case file produces exactly the typed error class declared in the manifest, or parses cleanly when expected. |

## What an honest "PASS" actually means

* **Strong claims**: M1.b, M1.e, M1.f, M1.g, M1.h, M1.i are absolute checks — passing them is real evidence.
* **Regression claims**: M1.a, M1.c are *regression* gates — they detect drift from a baseline, but the baseline itself was generated from the parser output. The first time the goldens are committed they pass trivially. Their value is preventing future drift.
* **Required human review (Exaware)**: see `docs/exaware-acceptance.md` — the Exaware reviewer must spot-check 3 specific items in the IR for the EVPN spec at end of W2. The numeric scorecard cannot replace this.
* **Reported only**: M1.d (anchor detection) is reported but not gated — it's an M2 metric, surfaced here for visibility.

## Updating goldens (regression contract)

Goldens are the regression contract. When a parser change is intentional and approved:

```
./modular_tools.sh golden_diff     # see what would change
./modular_tools.sh golden_update   # rewrite goldens (with confirmation)
```

Updating a golden = an explicit decision to redefine "correct." The diff must be reviewed before commit.

## Out of scope for M1

* Requirement identification (M2)
* Functional vs. non-functional classification (M2)
* AI test plan generation (M3)
* Test code generation (M4)
* Web UI (M5)
* OCR for scanned PDFs (the parser raises `UnsupportedScannedPDFError` and stops)
* Vertical-merge (`vMerge`) tracking in tables — `rowspan` is recorded as 1
